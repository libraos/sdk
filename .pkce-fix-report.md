# PKCE Insecure-Origin Crypto Fallback — Fix Report

## What changed

### `clients/typescript/src/auth/pkce.ts`

- Added `_nobleDigest()` — an async helper that dynamically imports `@noble/hashes/sha2.js`
  and runs `sha256()` to implement `subtle.digest("SHA-256", …)` without native `crypto.subtle`.
- Rewrote `resolveCrypto()` with three-branch logic:
  1. **Explicit injection** (`provided !== undefined`) → returned unchanged, no override.
  2. **Full Web Crypto** (`globalThis.crypto` has both `subtle` + `getRandomValues`) → returned as-is (HTTPS / localhost / Node 18+; unchanged existing behavior).
  3. **Insecure-origin fallback** (`getRandomValues` present, `subtle` absent) → returns a synthesised `CryptoProvider` using `getRandomValues.bind(g)` + the noble-sha256 `digest` shim.
  4. **No CSPRNG** → throws the existing error message (unchanged).
- Updated top doc-comment to describe the insecure-origin (HTTP) fallback.

### `clients/typescript/src/auth/pkce.test.ts` (new)

Seven new tests covering:
- RFC 7636 §4 S256 test vector via **native** Web Crypto.
- RFC 7636 §4 S256 test vector via **explicit fallback** `CryptoProvider` (noble sha256).
- RFC 7636 §4 S256 test vector via **insecure-origin `globalThis`** stub (no `subtle`).
- `generateCodeVerifier` length via explicit fallback provider.
- `generateCodeVerifier` length via insecure-origin `globalThis` stub.
- Explicit `CryptoProvider` is used as-is (spy verifies `digest` was called on the injected object).
- No CSPRNG (`globalThis.crypto = {}`) → throws matching `/No Web Crypto|CSPRNG/i`.

`globalThis.crypto` is restored in `afterEach` so stubs never leak between tests.

### `clients/typescript/package.json`

Added runtime dependency: `@noble/hashes` (pure-JS SHA-2 family, no native deps).

## Concerns

- `@noble/hashes/sha2.js` is dynamically imported inside `_nobleDigest` so it is only loaded when the insecure-origin path is actually taken; HTTPS deployments pay zero cost.
- The fallback only handles `"SHA-256"` — which is the sole algorithm PKCE uses. Any other algorithm throws clearly.
- `@noble/hashes` requires Node ≥ 20.19.0 per its own `engines` field; the SDK already targets Node ≥ 18. The fallback path is browser-only (insecure-origin HTTP), so this engine gap is not a practical concern, but it is worth noting if the SDK is run in older Node environments.
